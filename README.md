# Change Request Form System

ระบบจัดการแบบฟอร์มคำร้องขอการเปลี่ยนแปลง (Change Request) ที่ถูกออกแบบมาเพื่อรองรับกระบวนการขออนุมัติ ติดตามสถานะ และออกรายงานแบบครบวงจร

## 1. UX Goal (เป้าหมายประสบการณ์ผู้ใช้งาน)
เป้าหมายหลักของระบบคือการทำให้กระบวนการส่งคำร้องขอการเปลี่ยนแปลงมีความโปร่งใส รวดเร็ว และติดตามได้ง่าย ผู้ใช้ต้องสามารถกรอกข้อมูลได้โดยไม่สับสน และควรรู้สถานะของการดำเนินการในทุกขั้นตอนได้อย่างชัดเจน 

- **Clarity**: แบบฟอร์มมีความชัดเจน แบ่งหมวดหมู่ข้อมูลที่ต้องกรอก
- **Efficiency**: ลดเวลาในการเขียนและออกเอกสาร ด้วยระบบสร้าง PDF และการส่งอีเมลอัตโนมัติ
- **Visibility**: มีหน้าจอแสดงสถานะและรายงานสรุปที่ครอบคลุม

## 2. Components (องค์ประกอบของระบบ)
ระบบถูกแบ่งออกเป็นโมดูลหลักๆ ดังนี้:

- **Form Management**: 
  - `InsertForm.php`: สำหรับสร้างคำร้องขอใหม่
  - `UpdateForm_V2.php`: สำหรับแก้ไขและปรับปรุงข้อมูลคำร้องขอที่มีอยู่
  - `CopyForm.php`: คัดลอกแบบฟอร์มที่มีอยู่เพื่อลดเวลาการกรอกข้อมูลซ้ำซ้อน
- **Document Generation**:
  - `pdf_chi.php` / `pdf_cho.php`: ระบบดึงข้อมูลจากฐานข้อมูลเพื่อสร้างเอกสาร PDF (ส่งออกได้ทันที)
- **Reporting & Tracking**:
  - `WaitView.php` / `status_show.php`: แสดงสถานะและการรอคอยการอนุมัติ
  - `reportAll.php` / `reportRev.php` / `reportstatus.php`: ระบบรายงานเพื่อดูภาพรวมทั้งหมด
- **Core Workflow**:
  - `sendMail.php`: จัดการการส่งอีเมลแจ้งเตือนผู้ที่เกี่ยวข้อง
  - `setting_admin.php`: สำหรับผู้ดูแลระบบจัดการตั้งค่าพื้นฐาน

## 3. Motion & Interaction Approach (แนวคิดการโต้ตอบ)
ถึงแม้ระบบหลักจะเป็น PHP และอาศัยการโหลดหน้าเพจ (Page Reload) เป็นหลัก แต่สามารถประยุกต์ใช้แนวคิด Micro-interactions ควบคู่ไปกับ CSS/JS ได้:
- **Form Validation Feedback**: แสดงผลการกรอกข้อมูลผิดพลาด (Error State) ทันทีที่ผู้ใช้พิมพ์เสร็จในแต่ละช่อง
- **Loading State**: เมื่อกดส่งฟอร์ม (Submit) หรือออกรายงาน (Generate PDF) ควรมีสถานะโหลดเพื่อไม่ให้ผู้ใช้กดซ้ำ
- **Hover/Focus State**: บนปุ่มแอกชันต่างๆ (อนุมัติ, ปฏิเสธ, พิมพ์เอกสาร) มีการตอบสนองที่ชัดเจน

## 4. Implementation Steps (ขั้นตอนการทำงานของสถาปัตยกรรม)
1. **Routing & Home**: ผู้ใช้เข้าสู่ `index.php` และ `home.php` เพื่อดู Dashboard หรือเมนูหลัก
2. **Data Entry**: ผู้ใช้เข้าสู่กระบวนการผ่านฟอร์ม (`InsertForm`, `UpdateForm_V2`) คัดกรองข้อมูลฝั่งไคลเอนต์และเซิร์ฟเวอร์
3. **Database Processing**: ข้อมูลถูกส่งไปประมวลผลผ่าน `config/function.php` และจัดการการรับส่งผ่าน SQL Server / MySQL (ซึ่งถูกตั้งค่าใน `config/connection.php`)
4. **Notification**: หากสำเร็จ ระบบจะ trigger `sendMail.php` เพื่อแจ้งผู้อนุมัติ
5. **Report & Export**: ข้อมูลที่ถูกอนุมัติสามารถถูกดึงออกมาเป็น PDF ผ่านโมดูลส่งออกเอกสาร 

## 6. Trade-offs (ข้อแลกเปลี่ยนและข้อสังเกตเพิ่มเติม)
- **Security vs Convenience**: การเก็บ Log (`sn.log`) และการจัดการรหัสผ่านผู้ใช้งานในฐานข้อมูลอาจมีความเสี่ยง จึงต้องทำการซ่อนหรือแยกไฟล์ที่ไม่ปลอดภัยออกจากการบันทึก Git (`.gitignore`)
- **Monolithic Form**: การรวมตรรกะทั้งหมดไว้ในหน้าเดียวเช่น `UpdateForm_V2.php` (ประมาณ 189KB) ช่วยให้จัดการการส่งค่าง่าย แต่อาจทำให้การบำรุงรักษาในระยะยาวซับซ้อนขึ้น ควรพิจารณาแยก Component ในอนาคต
